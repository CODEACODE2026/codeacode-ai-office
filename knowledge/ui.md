# UI Knowledge

## Objetivo
Orientar padroes visuais reutilizaveis para interfaces da Code a Code.

UI deve materializar a UX aprovada e respeitar `DESIGN_PRINCIPLES.md`.

## Principios
- Interface deve parecer profissional e confiavel.
- Projetos web devem evitar aparencia de CRUD generico.
- Layout deve ser consistente com o tipo de produto.
- Elementos visuais devem ajudar a acao, nao competir com ela.
- Estados visuais devem ser claros.
- Responsividade nao e opcional.

## Aparencia desejada
Para sistemas, SaaS e dashboards:
- visual limpo;
- hierarquia clara;
- componentes consistentes;
- densidade equilibrada;
- cores funcionais;
- tipografia legivel;
- espacos previsiveis;
- foco em produtividade.

## Evitar
- cards excessivos sem funcao;
- gradientes decorativos sem motivo;
- tabelas quebradas no mobile;
- botoes desalinhados;
- formularios sem agrupamento;
- modais sem acao clara;
- textos longos dentro de botoes;
- paleta monotematica demais;
- hero de marketing em sistema operacional.

## Componentes essenciais
Todo design system interno deve cobrir:
- botoes;
- inputs;
- selects;
- checkboxes;
- tabs;
- filtros;
- tabelas;
- cards funcionais;
- modais;
- toasts;
- badges;
- empty states;
- loading/skeleton;
- navegacao.

## Estados obrigatorios
Componentes interativos devem prever:
- default;
- hover;
- focus;
- disabled;
- loading;
- error;
- success;
- selected.

## Layout
Boas praticas:
- usar grid consistente;
- controlar largura de conteudo;
- manter alinhamento vertical;
- reservar espaco para textos variaveis;
- evitar sobreposicao;
- usar overflow controlado em tabelas;
- testar desktop e mobile.

## Cores
Cores devem ter funcao:
- primaria para acao principal;
- verde para sucesso;
- amarelo para alerta;
- vermelho para erro ou destrutivo;
- cinzas para estrutura;
- cor ativa para navegacao.

Evitar depender apenas de cor para comunicar estado.

## Tipografia
Usar escala simples:
- titulo de pagina;
- titulo de secao;
- texto padrao;
- texto auxiliar;
- numeros ou indicadores.

Nao usar tamanho hero dentro de paineis densos.

## UI Review
Antes de aprovar interface, verificar:
- hierarquia;
- consistencia;
- responsividade;
- estados obrigatorios;
- acessibilidade basica;
- legibilidade;
- alinhamento;
- ausencia de sobreposicao;
- aderencia ao objetivo da tela.

## Codex
Ao pedir UI para Codex, enviar:
- direcao criativa aprovada;
- componentes esperados;
- estados obrigatorios;
- restricoes responsivas;
- arquivos permitidos;
- criterio visual de aceite.
