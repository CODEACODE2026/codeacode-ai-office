# Forms

## Objetivo
Formularios coletam dados com clareza, validacao e menor friccao possivel.

## Quando utilizar
Use formularios para:
- cadastro;
- edicao;
- filtros avancados;
- configuracoes;
- login;
- pagamentos;
- envio de contato.

## Quando evitar
Evite formulario longo quando:
- a acao puder ser dividida em etapas;
- poucos campos resolvem o fluxo;
- a informacao pode ser inferida;
- o usuario ainda nao tem contexto para preencher.

## Variacoes recomendadas
- Formulario simples.
- Formulario em etapas.
- Formulario compacto em painel.
- Filtro de tabela.
- Login.
- Modal de acao rapida.

## Boas praticas de UX
- Agrupar campos relacionados.
- Marcar obrigatorios.
- Validar cedo sem interromper demais.
- Mostrar erro perto do campo.
- Preservar dados quando houver erro.
- Evitar campos obrigatorios desnecessarios.

## Boas praticas de UI
- Labels sempre visiveis em sistemas.
- Inputs com altura consistente.
- Espacamento previsivel.
- Mensagens de erro legiveis.
- Campos monetarios e datas com formato claro.
- Acoes alinhadas ao fim do formulario.

## Acessibilidade basica
- Label associado ao campo.
- Foco visivel.
- Erro lido por tecnologia assistiva quando possivel.
- Ordem de tabulacao coerente.
- Placeholder nao substitui label.

## Erros comuns
- Formularios gigantes sem secao.
- Campo obrigatorio sem indicacao.
- Erro generico no topo.
- Botao salvar distante demais.
- Perder dados apos falha.
- Select apertado em mobile.

## Checklist
- campos obrigatorios estao claros?
- erros sao especificos?
- formulario funciona no mobile?
- dados sao preservados em falha?
- acao primaria esta visivel?
- campos estao agrupados?

## Reutilizacao pelos agentes
Analista define dados necessarios. UX Lead reduz friccao. Diretor Criativo UI define layout. DEV implementa validacao. QA testa erros e responsividade.
