# Dashboard

## Objetivo
Dashboard oferece visao rapida para decisao. Ele deve destacar indicadores, alertas e proximas acoes.

## Quando utilizar
Use dashboard quando o usuario precisa:
- acompanhar indicadores;
- identificar problemas;
- comparar periodos;
- ver alertas;
- acessar atalhos;
- tomar decisoes rapidas.

## Quando evitar
Evite dashboard quando:
- nao houver dados relevantes;
- a tela virar decoracao;
- graficos forem complexos sem necessidade;
- o usuario precisa primeiro executar tarefa operacional.

## Variacoes recomendadas
- Dashboard executivo.
- Dashboard operacional.
- Dashboard financeiro.
- Dashboard comercial.
- Dashboard de estoque.
- Painel de alertas.

## Boas praticas de UX
- Mostrar primeiro o que exige acao.
- Usar periodo claro.
- Indicar dados vazios.
- Destacar alertas criticos.
- Evitar excesso de metricas.
- Permitir caminho rapido para detalhe.

## Boas praticas de UI
- Cards pequenos e objetivos.
- Numeros legiveis.
- Cores funcionais.
- Graficos simples.
- Grid responsivo.
- Hierarquia clara entre resumo e detalhe.

## Acessibilidade basica
- Indicadores devem ter texto alem de cor.
- Graficos precisam de resumo textual quando possivel.
- Contraste adequado.
- Foco visivel em atalhos.

## Erros comuns
- Dashboard com cara de landing page.
- Grafico sem decisao associada.
- Cards grandes demais.
- Indicadores sem periodo.
- Dados vazios parecendo erro.
- Cor demais sem funcao.

## Checklist
- indicadores tem periodo?
- ha estado vazio?
- alertas estao claros?
- usuario sabe onde clicar?
- mobile esta legivel?
- graficos ajudam decisao?

## Reutilizacao pelos agentes
Analista define indicadores. UX Lead define prioridade. Diretor Criativo UI define composicao. DEV implementa dados e estados. QA valida consistencia.
